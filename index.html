Project Introduction

Multi-Voltage Macro Engine Architecture

The mv_macro_engine project is a high-performance multi-voltage ASIC design implemented in 32nm CMOS technology. It features a robust architecture comprising 76K standard cells and 100 hard macros, including 16 lp_ram_subsystem SRAMs and 84 default hard macros.

To optimize for power efficiency and meet strict SoC thermal constraints, the design employs a dual-voltage architecture based on aggressive voltage scaling. The logic is partitioned into two distinct domains:

Voltage Domains

The design is divided into two voltage domains:

  • High-Performance Domain (0.95V / VDD): This domain houses the primary computational engine and high-speed logic. It operates at 0.95V to ensure maximum throughput and timing closure, utilizing the saed32rvt_ss0p95v125c.db library.
  • Low-Power Domain (0.75V / VDDL): This domain incorporates the memory subsystems and low-speed control logic. By operating at 0.75V, the design significantly minimizes both dynamic switching power and static leakage. It utilizes the saed32rvt_ss0p75v125c.db library, optimized for low-voltage operation.

Physical Design Challenges

The multi-voltage architecture of the mv_macro_engine introduces several sophisticated backend design requirements:

  • UPF-based Power Intent Integration: Seamless integration of logical power intent with the physical design database.
  • Level Shifter Management: Ensuring signal integrity for all crossings between the 0.75V and 0.95V domains using dedicated ulvl and dlvl level shifters.
  • Macro-Centric Floorplanning: Strategic placement of 100 hard macros, including memory subsystems, while maintaining routability and minimizing wire detours.
  • Dedicated Voltage Areas & PDN: Implementation of a multi-domain mesh Power Delivery Network (PDN) featuring VDDL guard bands for the low-power engine.
  • H-Tree Clock Distribution: Managing an H-Tree clock topology to minimize skew while handling voltage-level crossings at the clock spine.
  • MCMM Timing Analysis: Balancing timing closure across complex multi-corner multi-mode scenarios to ensure robustness across process, voltage, and temperature variations.

RTL - Multi-Voltage Macro Engine

Designed a 128-bit cryptographic accelerator using a multi-voltage architecture to optimize both power consumption and performance in ASIC-based systems. The design separates functionality into a 0.75V low-power domain for key generation and a 0.95V high-performance domain for encryption processing, demonstrating practical power-aware digital design techniques.

Architecture Overview

The low-power block generates a pseudo-random keystream using a 128-bit LFSR-based key scheduler, while the high-performance encryption engine implements a 64-round Substitution-Permutation Network (SPN) architecture. A dedicated voltage-domain interface safely transfers key data between domains, reflecting real-world ASIC design practices for power-domain integration.

Key Features

  • 128-bit encryption datapath with configurable architecture
  • Low-power key generation operating at 0.75V
  • High-performance cipher core operating at 0.95V
  • Safe voltage-domain crossing through interface registers
  • 64-round SPN-based encryption engine
  • Parallel 4-bit S-Box implementation for non-linear transformations
  • Deep pipelining for improved throughput and timing performance
  • Fully synthesizable and modular Verilog RTL

Applications

Secure SoCs, IoT security accelerators, embedded cryptographic engines, hardware security modules (HSMs), and power-constrained ASIC designs.

Level shifter strategies were also specified to safely manage signal transfers between voltage domains, addressing voltage compatibility requirements during physical design.

Clock Tree Synthesis (CTS)

Developed a complete Clock Tree Synthesis (CTS) execution script for the mv_macro_engine SoC, implementing a high-precision H-Tree clock topology. The flow focuses on balanced distribution, skew control, and timing closure across multiple PVT corners. The implementation integrates clock constraints, NDR rules, clock uncertainty management, and multi-voltage level shifting to ensure robust clock propagation across the 0.95V and 0.75V domains.

Key Features: H-Tree & CTS Strategy

  • H-Tree Topology: Implemented a symmetrical H-Tree structure to guarantee uniform insertion delay and minimal skew across the high-performance cryptographic datapath.
  • Standard-Cell-Only Tree: Focused CTS flow on the standard-cell clock network, with balanced distribution to macro clock pins.
  • Clock-Specific Routing (NDR): Defined custom Non-Default Rules (NDR) using double-width and double-spacing to minimize cross-talk.
  • Multi-Voltage Level Shifting: Integrated clock-domain level shifters specifically for the transition into the 0.75V domain, ensuring timing-accurate clock edges.
  • CRPR & Uncertainty: Enabled Clock Reconvergence Pessimism Removal (CRPR) and modeled multi-corner setup/hold uncertainty for robust signoff.
  • DFT-Aware: Configured scan-clock structures to ensure seamless integration between functional and test modes.

Flow Breakdown

1. Clock Routing Rules (NDR)

  • Removes default routing constraints for clock nets.
  • Defines double-width & double-spacing rules (NDR) to mitigate signal integrity risks.
  • Layer Constraint: Restricts clock trunk routing to M5/M6 layers to preserve M7/M8 for the power delivery mesh.

2. H-Tree & Constraint Definition

  • H-Tree Topology: Forces perfectly symmetrical wire lengths to all quadrants, critical for the high-frequency mv_macro_engine datapaths.
  • Skew Targets: Strict skew targets per corner: SS corners (relaxed) and FF corners (tight: 0.02ns).
  • Transition Limits: Set at 0.15 ns to maintain edge sharpness.

3. CTS Execution Flow

  • Build Clock Tree: Generates the balanced H-Tree structure based on physical macro/port coordinates.
  • Clock Routing: Physically routes the H-Tree backbone and leaf-level nets.
  • Multi-Voltage Level Shifter Insertion: Automatic insertion of level shifters at domain crossings within the clock network.

4. Final Optimization Stage

  • Performs high-effort hold fixing using High-VT (HVT) delay cells to minimize leakage.
  • Final skew cleanup and DFT-aware scan clock optimization.
  • Produces a signoff-ready clock network compliant with PrimeTime analysis.

Setup and Hold Fixes

Achieving timing closure for the mv_macro_engine requires rigorous setup and hold optimization across multiple PVT corners in the MCMM environment.

⚠️ Note: Both insert_buffer and upsize_cell manual scripts are not recommended in the standard ICC2 flow. For production silicon, always rely on native route_opt or PrimeTime DMSA ECO generation to resolve timing violations, as they account for accurate 3D SPEF parasitics and complex routing design rules.

Available Fixes for Setup Violations

Setup violations occur when the data path is too slow. The following optimization techniques are available to speed up the data arrival time:

  • Cell Upsizing: Replacing a driver with a higher drive-strength standard cell to push the capacitive load faster.
  • Buffer Insertion: Breaking long, highly capacitive nets by inserting high-drive buffers to improve signal transition.
  • VT Swapping (LVT): Swapping High-VT or Regular-VT cells with Low-VT (LVT) variants for faster switching speed (at the cost of higher leakage power).
  • Path Restructuring: Cloning logic or pulling standard cells closer together during placement to reduce wire delay.
  • Automated Flow: Using route_opt in ICC2 or executing a targeted fix_eco_timing -setup script exported from PrimeTime.
View Upsize Cell Script on GitHub View Insert Buffer Script on GitHub

Available Fixes for Hold Violations

Hold violations occur when the data path is too fast, causing data to be prematurely overwritten at the capturing flip-flop. The following techniques are used to slow down the data path:

  • Delay Buffer Insertion: Adding specialized delay cells or buffers (e.g., DELLN*_HVT) directly into the fast data path to intentionally increase delay without affecting logic.
  • Cell Downsizing: Reducing the drive strength of a cell to intentionally slow down its transition and delay time.
  • VT Swapping (HVT): Swapping LVT/RVT cells for High-VT (HVT) cells to slow down the path, which also provides the benefit of saving leakage power.
  • Automated Flow: Using route_opt -hold in ICC2 or executing a targeted fix_eco_timing -hold script exported from PrimeTime (usually performed only after CTS and detailed routing).

Successfully closed timing for the mv_macro_engine by utilizing PrimeTime-driven ECOs for both setup and hold paths, ensuring zero setup violations across all functional/scan modes and achieving a pristine, silicon-ready signoff.

Timing Analysis: PrimeTime Signoff

This section details the critical Static Timing Analysis (STA) and signoff flow used to ensure the mv_macro_engine meets all performance targets across multiple operating scenarios.

1. Post-Route Integrity Checks

Before initiating signoff, the routed design must be clean:

  • DRC Cleanup: Fix all shorts, opens, and floating wires using check_pg_drc -do_not_check_in_hier_blocks.
  • LVS Verification: Validate netlist vs. layout with check_lvs -max_errors 0.
  • Route Verification: Confirm completion of all nets with check_routes.

2. StarRC Parasitic Extraction

Extraction of RC parasitics is performed to feed accurate data into PrimeTime:

  • Workspace: mkdir -p STARRC/{inputs,outputs/{spef,work},scripts}
  • Extraction: Use the dedicated scripts in the STARRC directory.
  • Output: Generates cbest.spef (for fast corner) and cworst.spef (for slow corner).
  • Handoff: Copy extracted SPEF files to PRIMETIME/inputs/.

3. PrimeTime Signoff Flow

Workspace Setup

  • Initialize: mkdir -p PRIMETIME/{inputs,outputs,scripts,sessions}
  • Read Routed Netlist: write_verilog ./PRIMETIME/inputs/routed_netlist.v
  • Export post-route SDCs (avoid ideal clocks):
    • Setup: write_sdc -scenarios func.ss_125c -output ./PRIMETIME/ss_125_clean.sdc
    • Hold: write_sdc -scenarios func.ff_m40c -output ./PRIMETIME/ff_m40_clean.sdc

Execution Stages

  • Session Generation: Invoke pt_shell. Source func_max.tcl and func_min.tcl sessions independently. Note: Ensure link_library contains only 0.95V base cells, SRAMs, and Level Shifters to avoid scaling group crashes (SLG-303).
  • DMSA Execution: Invoke pt_shell -multi_scenario and execute the master dmsa.tcl script to close timing across all scenarios.
  • ECO Handoff: Export fixes using write_changes -format icc2tcl -output ./outputs/Multi_scenario_final_eco.tcl.

4. Final Implementation & Waivers

After sourcing the ECO file in icc2_shell:

  • Incremental Fixes: Run legalize_placement -incremental, route_eco, and final DRC/LVS cleanup.
  • Waivers:
    • Hold Violations: ~5 fs hold violations are treated as OCV noise and waived, as forcing X2 buffers would break setup timing.
    • DRVs (Max_trans/cap): ICC2 TLU+ pessimism is overridden by PrimeTime 3D SPEF accuracy. Waive ICC2 DRVs in favor of PrimeTime signoff.

MCMM Constraints - Multi-Corner Multi-Mode

MCMM (Multi-Corner Multi-Mode) timing analysis is critical for ensuring robust physical design closure across multiple operating conditions. This section provides comprehensive details on the MCMM setup and constraint methodology used in the crypto_top project.

MCMM Overview

Multi-Corner Multi-Mode analysis ensures that the design meets timing requirements across:

  • Process Corners: FF (Fast-Fast) and SS (Slow-Slow)
  • Temperature Conditions: -40°C and 125°C
  • Operating Modes: Functional and Scan/Test
  • Voltage Domains: 0.95V and 0.75V

Corner Specifications

Corner Process Temperature Library Use Case
FF (-40°C) Fast-Fast -40°C saed32rvt_ss0p95v_m40c.db Best-case hold analysis
SS (125°C) Slow-Slow 125°C saed32rvt_ss0p95v125c.db Worst-case setup analysis

Operating Modes

Functional Mode

  • Normal operational mode
  • Encryption/decryption operations
  • Core cryptographic functions active

Scan/Test Mode

  • Test/Debug operational mode
  • Scan chain enabled
  • Different timing characteristics due to scan path

MCMM Analysis Benefits

  • Comprehensive timing coverage across all operating conditions
  • Detects timing failures that single-corner analysis might miss
  • Ensures hold time compliance in best-case conditions
  • Ensures setup time compliance in worst-case conditions
  • Enables efficient multi-voltage timing closure
  • Reduces silicon failure risk and improves reliability

Constraint Management in Physical Design

During physical design stages (placement, CTS, routing), MCMM constraints guide:

  • Placement Optimization: Cell placement to minimize critical path delays
  • Clock Tree Synthesis: Clock skew control across all corners
  • Routing: Wire routing prioritizing critical timing paths
  • Signoff Analysis: Final timing verification across all scenarios
